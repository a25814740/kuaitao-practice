# 塊陶啊!!｜Vue 3 求職練功專案規格

## 1. 專案目的
這是一個用來練習 Vue 3 的 side project。

使用者背景：
- 有 8 年 HTML / CSS / JavaScript / jQuery / Vue 2 基礎
- 約 2 年未實際開發，想重新找回手感
- 求職目標以 Vue 3 職缺為主
- 不想用純看影片的方式學習，希望透過「做出有趣作品」學 Vue 3
- 希望能從簡單到困難，逐步完成一個完整 App
- 每個階段都要看得到學習進度、成果差異，方便之後複習

此專案不是一開始就追求完整商業化，而是以「學習路徑清楚、每階段可驗收、適合作品集展示」為優先。

---

## 2. 專案概念
專案名稱暫定：**塊陶啊!!**

概念：
- 類似「薪水小偷」的情緒型 App
- 核心不是摸魚，而是記錄「下班那一刻」
- 使用者可以記錄下班、加班、排行、好友互動、貼文抱怨
- 台灣市場可用「塊陶啊!!」這個諧音梗
- 其他中文市場可替換為「快逃啊」

產品核心價值：
- 儀式感：下班那一瞬間變成可記錄、可炫耀的事件
- 情緒共鳴：對加班文化的幽默反抗
- 遊戲化：排行、成就、稱號、等級、任務
- 社交性：好友互比、貼文互動、行業/地區排行

### 長期版本

## 4. 核心功能清單

### 4.1 使用者初始化設定
第一次進入 App 需先設定：
- 暱稱
- 大頭貼
- 工作類型（先固定幾種）
- 每天下班時間
- 收入

備註：
- 工作類型先寫死：例如工程師、設計師、行銷、客服、餐飲、行政、公職、醫療、其他
- 大頭貼一開始可先用預設圖或 emoji / 頭像字母
- 收入後續可用於推導時薪、加班感受、戰績數據

### 4.2 首頁
- 顯示目前時間
- 顯示距離下班還有多久
- 顯示今日狀態（未到下班 / 已到下班 / 已加班）
- 顯示一個主按鈕：下班點我 / 塊陶啊!!
- 點擊後記錄當下時間
- 記錄後顯示結果卡片

### 4.3 下班記錄
- 記錄按下按鈕的實際時間
- 顯示與預設下班時間差距
- 可判斷：
  - 準時下班
  - 提早下班
  - 延後下班
- MVP 先精確到毫秒即可
- UI 可以營造微秒感，但後端與資料記錄不必真的追求奈秒

### 4.4 排行榜
- 可先用假資料 / 本地資料實作 UI 與排序邏輯
- 分類：
  - 好友
  - 全國
  - 地區
  - 行業
- 之後再串 Supabase

### 4.5 社交牆
- 使用者可發簡短抱怨 / 心得
- 其他人可留言
- MVP 可先做假資料版
- 第二階段再做真實資料庫版

### 4.6 遊戲化
- 等級
- 成就
- 稱號
- 任務

成就範例：
- 連續 7 天準時下班
- 連續 3 天加班超過 2 小時
- 第一次在下班時間後 1 秒內點擊

## 6. 分階段學習與開發路線

> 核心原則：每一階段都要「能跑、有成果、可回顧、可展示」。

### Stage 1｜Vue 3 基礎響應式
對應主題：
- `<script setup>`
- `ref`
- 事件綁定
- 基本模板語法

目標：
- 做出一個首頁按鈕
- 點擊後記錄當下時間
- 顯示最後一次下班時間

完成標準：
- 可成功點擊
- 畫面即時更新
- 可理解 Vue 3 最基礎資料流

產出：
- 單頁雛形
- 按鈕 + 結果顯示

建議拆分：
- Step 1：先把 `App.vue` 預設內容清掉，只留下自己的標題、說明文字與主容器。
- Step 2：建立 `ref` 狀態，例如 `lastEscapeTime`、`clickCount`。
- Step 3：建立 `handleEscape` 函式，點擊後記錄目前時間。
- Step 4：把時間與次數渲染到畫面上，確認畫面會即時更新。
- Step 5：補上簡單版型與按鈕樣式，讓首頁有基本展示感。
- Step 6：確認 `npm run dev` 可正常啟動，再更新 `DEVLOG` 與 `STAGE_CHECKLIST`。

學習重點：
- 先理解 Vue 3 最基本的資料流：資料改變，畫面跟著改變。
- 先把單檔元件寫熟，不急著拆元件。
- 這一階段只追求最小可運作版本，不追求完整功能。

---

### Stage 2｜條件渲染與計算邏輯
對應主題：
- `computed`
- `watch`
- `v-if`
- 倒數與狀態切換

目標：
- 加入固定下班時間
- 顯示距離下班還剩多久
- 根據時間切換按鈕狀態

完成標準：
- 還沒到下班時間時顯示不可逃跑狀態
- 到了下班時間後顯示「塊陶啊!!」
- 點擊後可顯示差距時間

產出：
- 有狀態感的首頁
- 基礎倒數邏輯

建議拆分：
- Step 1：先寫死一個下班時間，例如 `18:00`。
- Step 2：建立目前時間狀態，並用計時器每秒更新。
- Step 3：用 `computed` 算出距離下班還有多久。
- Step 4：用 `computed` 或 `watch` 判斷現在是未下班、已下班、已加班。
- Step 5：用 `v-if` / `v-else` 切換文案、按鈕狀態與提示區塊。
- Step 6：點擊後補上「比預定時間早/晚多久」的結果顯示。
- Step 7：整理條件邏輯，避免 `template` 裡塞太多判斷。

學習重點：
- 練習把「原始資料」轉成「畫面需要的資料」。
- 開始建立時間狀態與條件渲染的心智模型。
- 學習什麼邏輯該放 `computed`，什麼情況才需要 `watch`。

---

### Stage 3｜元件化
對應主題：
- `defineProps`
- `defineEmits`
- component 拆分

目標：
將首頁拆為：
- EscapeButton.vue
- TimeDisplay.vue
- ResultCard.vue
- RecordHistory.vue

完成標準：
- 主頁不再塞滿所有邏輯
- 能理解父子元件資料流

產出：
- 更像正式專案的結構
- 可複用元件

建議拆分：
- Step 1：先盤點首頁目前有哪些區塊，決定哪些適合拆出去。
- Step 2：先拆純展示型元件，例如 `TimeDisplay.vue`、`ResultCard.vue`。
- Step 3：再拆帶互動的 `EscapeButton.vue`。
- Step 4：把歷史紀錄區塊拆成 `RecordHistory.vue`。
- Step 5：用 `defineProps` 傳資料給子元件。
- Step 6：用 `defineEmits` 讓按鈕點擊事件往父層回傳。
- Step 7：整理 `App.vue`，讓它只保留頁面整合邏輯。

學習重點：
- 理解元件拆分不是為了炫技，而是讓職責更清楚。
- 練習父傳子、子傳父的基本資料流。
- 建立「畫面區塊」與「邏輯責任」的拆分概念。

---

### Stage 4｜Composables 與邏輯抽離
對應主題：
- composables
- `onMounted`
- `onUnmounted`
- `requestAnimationFrame`

目標：
- 把時間計算抽成 `useEscapeTimer`
- 把下班判斷抽成 `useOffDutyStatus`

完成標準：
- UI 與邏輯明顯分離
- 邏輯可重複使用

產出：
- composables 架構
- 更符合 Vue 3 實戰寫法

建議拆分：
- Step 1：先找出哪些邏輯和畫面無關，例如現在時間更新、倒數計算、狀態判斷。
- Step 2：建立 `useEscapeTimer`，負責現在時間與倒數資訊。
- Step 3：建立 `useOffDutyStatus`，負責判斷是否下班、是否加班。
- Step 4：把原本寫在 `App.vue` 或頁面裡的邏輯搬進 composables。
- Step 5：在元件中重新接回 composables 回傳的狀態與方法。
- Step 6：確認畫面行為不變，代表抽離成功。
- Step 7：在 `DEVLOG` 寫清楚哪些邏輯為什麼適合抽成 composable。

學習重點：
- 理解 composable 是為了重用與分離邏輯，不只是把檔案搬家。
- 練習生命週期與計時器清理。
- 開始建立「UI 層」和「邏輯層」的分工。

---

### Stage 5｜Pinia 全域狀態
對應主題：
- store
- 使用者資料管理
- 跨頁共享狀態

目標：
- 建立 user store
- 設定頁可修改暱稱、大頭貼、職業、下班時間、收入
- 首頁可即時讀取設定

完成標準：
- 多頁共享同一份資料
- 可理解 Pinia 為何取代 Vuex

產出：
- 設定頁
- 個人化首頁

建議拆分：
- Step 1：先定義哪些資料需要跨頁共享，例如暱稱、職業、下班時間、收入。
- Step 2：建立 `user` store，先用最小欄位開始。
- Step 3：把設定頁表單與 store 綁起來。
- Step 4：首頁讀取 store 資料並顯示個人化內容。
- Step 5：如果需要，補上預設值與簡單資料驗證。
- Step 6：確認重新整理前的資料流與畫面顯示都正常。
- Step 7：整理哪些狀態該留在元件內，哪些該進 store。

學習重點：
- 練習區分 local state 與 global state。
- 理解 Pinia 在 Vue 3 專案中的定位。
- 學習從「單頁互動」轉向「多頁共享資料」。

---

### Stage 6｜Vue Router 與多頁面

## 7. 時間精度設計原則

### UI 層
- 可以顯示到毫秒
- 可以用動畫營造微秒感
- 用 `performance.now()` 增強畫面表現

### 資料層
- MVP 只需記錄到毫秒即可
- 因為網路延遲與裝置時間差，真正追求微秒或奈秒在產品上意義不大
- 後端儲存重點是「使用者按下當下的 client timestamp」與「預設下班時間差值」

結論：
- 微秒主要是儀式感與 UX
- 不必一開始就把資料庫與 API 設計成高精密科學系統

## 9. 給 Codex 的執行原則

Codex 在此專案中應遵守：

1. 先以「學習型專案」為目標，不要一開始過度工程化。
2. 優先讓每個 stage 都有可執行成果。
3. 每完成一個 stage，必須更新：
   - `docs/STAGE_CHECKLIST.md`
   - `docs/DEVLOG.md`
4. 每次修改都要說明：
   - 本次學到的 Vue 3 概念
   - 與前一階段差異
   - 本階段完成內容
   - 下一階段建議
5. 優先保留簡潔架構，不要太早引入不必要的抽象。
6. 如果需求尚未進入 Supabase 階段，就先用本地假資料。
7. UI 可先簡潔，但需具備作品集展示感。
8. 優先使用 TypeScript 與否可視使用者當前學習負擔決定；若會增加太多阻力，可先用 JavaScript 版本完成，再逐步轉型。

## 12. 後續給 Codex 的工作方式

之後每次只下達一種指令：

### 模式 A：請做下一階段
